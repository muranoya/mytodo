# mytodo — プロジェクト方針

このドキュメントは **意図・設計思想・運用ルール**だけを扱う。
パッケージ名・SDK バージョン・ディレクトリ構成・ドメインモデルのフィールド・依存ライブラリのバージョン等、コードを読めば分かる事実は記載しない（更新忘れによるドキュメント負債を避けるため）。一次情報は常にソースコード／`build.gradle.kts` / `libs.versions.toml` / `strings.xml` を参照すること。

## 1. プロジェクト概要

Android ネイティブの TODO アプリ。単一ユーザー・オフライン動作・片手操作前提のモダン UI を目指す。

### 機能要件

- TODO の一覧 / 作成 / 完了 / 更新 / 削除
- フォルダ: TODO は必ずいずれかのフォルダに属する
- 優先度: システムが規定する**固定の優先度集合**から選択する。ユーザーによるラベル追加・色のカスタマイズは提供しない（選択コストを下げ、UI を一貫させるため）
- ゴミ箱: 完了・削除した TODO を 30 日保持し、経過後に物理削除
- 並び替え: TODO / フォルダともに長押し＋ドラッグでユーザー任意に並び替え可能
- **アクティブ一覧画面**: 全フォルダ横断で未完了・未削除の TODO を優先度順に表示する専用画面を用意する（フォルダ単位の画面とは別に、トップレベルで切り替え可能）

### 非機能要件

- 端末ローカルのみ（同期なし）。将来のクラウド同期に備え Repository 層で抽象化は保つ
- 片手操作最優先。親指導線上に主要アクションを配置する
- モダンでクールな UI（Material 3 + Dynamic Color）

## 2. 技術スタック

Android 標準のモダン構成を採用する。具体バージョンは `gradle/libs.versions.toml` を一次情報とする。

| 領域 | 採用 |
| --- | --- |
| 言語 | Kotlin |
| UI | Jetpack Compose + Material 3（Expressive motion 有効） |
| ビルド | Gradle Kotlin DSL + Version Catalog |
| 永続化 | Room（KSP 経由） |
| DI | Hilt |
| 非同期 | Kotlin Coroutines + Flow（UI state は `StateFlow`） |
| ナビゲーション | Navigation Compose（type-safe routes） |
| バックグラウンド | WorkManager（ゴミ箱 30 日経過品の物理削除） |
| テスト | JUnit / Compose UI Test、および Room の Migration テスト |

### バージョン指定ルール

- すべての依存バージョンは Version Catalog に一元管理し、`*.gradle.kts` からは alias 参照のみ（ハードコード禁止）
- `gradle.lockfile` を有効化してコミットする
- 依存更新は明示的な操作で行う（自動アップデートに任せない）

## 3. アーキテクチャ

- レイヤー分離: `ui` / `domain` / `data` の 3 層を基本とし、各機能・画面ごとにサブパッケージを切る
- 単方向データフロー (UDF): ViewModel が `StateFlow<UiState>` を公開し、Composable は受動描画
- ViewModel は Android フレームワーク非依存に保つ（`Context` / `Resources` を直接扱わず、`@StringRes` や domain 層の列挙で返す）
- Repository はインタフェースを `domain` 層に置き、実装を `data` 層に置く（将来のクラウド同期実装を差し込めるように）

## 4. ドメインモデル

### Folder / Todo

- `Folder` は表示順を持ち、TODO は必ずいずれかのフォルダに属する
- `Todo` は完了状態と論理削除状態を保持し、ゴミ箱 TTL を計算するために**完了時刻**と**削除時刻**のタイムスタンプを持つ

### Priority

- 優先度は**ソート順（rank）と表示ラベル**を持つ固定集合として表現する。ラベル文言は `strings.xml` を一次情報とする
- **表示色は `ui/theme` 側で一元管理**し、TODO や Priority 自体には色属性を持たせない（テーマ切替で配色を一括制御するため）
- 永続化はソート順を整数として保存し、SQL で `ORDER BY` 可能にする

### ゴミ箱

- 論理削除: 完了済みまたは削除済み（タイムスタンプが入っているもの）を「ゴミ箱内」とみなす
- 物理削除: 完了時刻または削除時刻から 30 日経過した行を WorkManager の定期ジョブで削除する
- Doze モード下でも実行されやすいよう深夜帯に寄せる

## 5. UI / UX 原則

### 片手操作

- 主要アクションは画面下部に集約し、親指で届く範囲に置く
- 編集 UI は画面上部に被せず、**下から展開するモーダルシート**で開く
- 新規作成も同じシートからインライン入力で完結させ、画面遷移を最小化する
- 画面上部の見出し領域は装飾に留め、操作要素を置かない

### ジェスチャ

- 長押し＋ドラッグで並び替え（フォルダ別一覧のみ有効）
- 右スワイプで完了、左スワイプで削除
- 完了・削除はスナックバーから即 Undo 可能にする（誤操作防止）

### 画面構成

- トップレベルは **アクティブ横断 / フォルダ別 / ゴミ箱** の 3 つで切り替え可能にする
- アクティブ横断画面:
  - 全フォルダの未完了かつ未削除の TODO を表示
  - 優先度バケットでセクション分け、セクション内は作成日時の昇順
  - ドラッグ並び替えは無効、各アイテムには所属フォルダ名をサブテキストで表示
  - スワイプでの完了／削除、タップでの編集はフォルダ別画面と同じ操作で揃える

### ビジュアル

- Material 3 Dynamic Color を既定で有効化、Android 12+ ではユーザー壁紙に追随する
- ダークモードファースト。色は `MaterialTheme.colorScheme` からのみ引き、リテラル Color の直書きは避ける
- タイポグラフィは Material 3 type scale を採用
- 操作フィードバックに HapticFeedback（完了・削除・並び替え確定で軽く鳴らす）
- モーションは Material 3 Expressive のスプリング系を基本とする

## 6. コーディング規約

- Kotlin 公式スタイルに準拠し、**静的解析・フォーマッタ（ktlint / Spotless 等）を CI ゲートに組み込む**
- Composable は状態ホイストを徹底し、公開 Composable には `@Preview` を最低 1 個用意する
- ViewModel は副作用を Flow 経由で一方向に流す（ViewModel から直接 Navigation しない）
- Room のスキーマ変更は必ず `Migration` を記述し、Migration テストを追加する
- 例外を握り潰さない。失敗は UI state に載せて画面に表現する

## 7. リポジトリ運用

- git 操作（`add` / `commit` / `push`）はユーザーの明示指示があるまで自動で行わない（グローバル方針準拠）
- PR タイトルは日本語で簡潔に。PR 本文には「意図」と「検証手順」を含める
- ブランチ名は `feature/<topic>`, `fix/<topic>` 等のプレフィックス

### 開発コマンドは just に集約する

テスト・Lint・ビルド・エミュレータ起動・APK インストール等の**開発作業コマンドはすべて `justfile` に集約する**。`./gradlew` や `adb` を直接叩くことは避け、新しい作業フローが必要になったら `justfile` にレシピを追加する。

- 利用可能なレシピ一覧は `just`（引数なし）で確認できる
- CI の品質ゲートも `just` レシピ越しに呼び出し、ローカルと CI で同じ手順が走るようにする
- これにより「どのコマンドで何が動くか」を 1 ファイルで一覧でき、コマンド表記の揺れも防げる
